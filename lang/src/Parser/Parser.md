# top down parser

• Distinction between language and grammar
– Language: set of strings over some alphabet
– Grammar: a set of rules for generating the strings in a language
– G: grammar, L(G): language generated by grammar

• Recognition vs. parsing, given grammar G and word w
– Recognition is decision problem - is w in L(G)?
– Parsing: if w in L(G), show a derivation (proof)

• Context-free grammar: 4-tuple
– S: Start symbol
– T: Terminals aka tokens (also written as Σ by some authors)
– N: Non-terminals (also written as V)
– P: Productions

• Sentential forms and sentences
– Sentential form: string that can be obtained by starting with S and
using productions as rewrite rules to rewrite nonterminals
– Sentence: sentential form without nonterminals (word in language)

• Derivation of string using grammar
– Start from S and repeatedly rewrite a nonterminal
using the productions of the grammar until there
are no nonterminals left
– Leftmost/rightmost-derivation: rewrite only the
leftmost/rightmost nonterminal at each step

• Ambiguous grammar
– Grammar in which there are two or more leftmost
derivations for some word

# Avoiding Ambiguous Grammar Rules

## 1. Use Precedence Levels for Expressions

Instead of one flat rule like `E → E + E | E * E | int` (which is ambiguous — `2 + 3 * 4` has two parse trees), split it into layers:

```
E  → T ('+' T)*
T  → F ('*' F)*
F  → int | '(' E ')'
```

Each layer handles one precedence level. Lower in the hierarchy = tighter binding. This forces `2 + 3 * 4` to parse as `2 + (3 * 4)` because `*` lives deeper in the grammar.

## 2. Enforce Associativity Through Rule Structure

Left-recursive rules give left associativity, right-recursive gives right. Since SLL parsers can't handle left recursion directly, use iteration (the `*` above) and build the AST left-to-right in your code.

Each layer handles one precedence level. Rules closer to the start symbol (E) have lower precedence, while rules further down (F) have higher precedence — meaning they bind tighter. This forces 2 + 3 * 4 to parse as 2 + (3 * 4) because * is in the T rule which is deeper in the grammar than + in the E rule, so * binds its operands first.

## 3. One Production Per Lookahead Token

When two productions for the same non-terminal have overlapping FIRST sets, left-factor:

```
// Ambiguous — both start with 'id'
Stmt → id '=' Expr
Stmt → id '(' Args ')'

// Left-factored — unambiguous
Stmt → id StmtTail
StmtTail → '=' Expr
StmtTail → '(' Args ')'
```

## 4. Use Keywords to Distinguish Constructs

Monkey does this well — `let`, `if`, `fn`, `return` each unambiguously signal which production to use. That's free disambiguation.

## 5. The Practical Test

Compute your FIRST and FOLLOW sets and build the parsing table. If any cell has more than one entry, your grammar is ambiguous (for SLL(1)). The conflict tells you exactly _where_ the ambiguity lives, then apply the techniques above to fix it.

**Recommended order:** Start with statements (keywords disambiguate easily), then tackle expressions with precedence layering.

The top-down construction of a parse tree is done
by starting with the ro ot, lab eled with the starting nonterminal stmt, and re-
p eatedly p erforming the following two steps.

1. At no de N , lab eled with nonterminal A, select one of the pro ductions for
   A and construct children at N for the symb ols in the pro duction b o dy.
2. Find the next no de at which a subtree is to b e constructed, typically the

Top-down parsing is the main focus of 2.4. The idea is to start from the start symbol of the grammar and try to "expand" it — applying production rules — until you match the actual input. You're building the parse tree from the root down to the leaves. The key technique introduced is recursive-descent parsing, where each nonterminal in the grammar gets its own function/procedure. When you call the function for a nonterminal, it looks at the next input token to decide which production rule to apply, then calls the functions for the symbols on the right-hand side of that rule.
Predictive parsing is a special case where you can always decide which production to use just by looking at the next token (one token of "lookahead"). This only works for certain grammars — ones where, for any nonterminal, the possible right-hand sides start with different tokens so there's no ambiguity about which to pick. Grammars that allow this are sometimes called LL(1) grammars.
The problem of left recursion: A rule like A → A α | β causes infinite recursion in a top-down parser because the function for A would immediately call itself forever. The section shows how to eliminate left recursion by rewriting it as A → β A' and A' → α A' | ε, which turns left recursion into right recursion — something a recursive-descent parser can handle.
Connecting it all together: The section is essentially showing you how to take a grammar, clean it up (remove left recursion, etc.), and then write a simple parser as a collection of mutually recursive functions — one per nonterminal — that consume input tokens left to right and build up the parse tree.

For a grammar to be LL(1), whenever you have a nonterminal with multiple productions like:
A → α | β | γ
the FIRST sets of α, β, and γ must be disjoint — no token can appear in more than one of them. That's what lets the parser look at just one token of lookahead and know exactly which production to pick, with no ambiguity.

There's one wrinkle though: if any of those right-hand sides can derive the empty string (ε), then you also need to bring in the FOLLOW set. Specifically, if α can derive ε, then FIRST(α) (minus ε) and FOLLOW(A) must also be disjoint from the FIRST sets of the other alternatives. The optexpr() function is a small example of this — it has two choices (match expr, or do nothing/ε). When it sees something that's not expr, it knows to take the ε path, because whatever it's seeing must be in the FOLLOW set of optexpr (like ; or )).
So the full LL(1) condition for A → α | β is:

FIRST(α) ∩ FIRST(β) = ∅
If α ⇒* ε, then FIRST(β) ∩ FOLLOW(A) = ∅ (and vice versa)

# Parse Tree vs Abstract Syntax Tree (AST)

## Parse Tree (Concrete Syntax Tree)

A parse tree is a direct, literal representation of how a grammar derives a string. Every rule application is visible in the tree.

**Properties:**

- Every internal node is a **nonterminal** from the grammar.
- Every leaf node is a **terminal** (token) from the input.
- The structure mirrors the grammar exactly — one node per production applied.
- Includes all syntactic detail: parentheses, semicolons, commas, keywords used only for disambiguation.

**Example:** For the grammar:

```
expr   → expr + term | term
term   → term * factor | factor
factor → ( expr ) | id
```

Parsing `a + b * c` produces:

```
            expr
           / | \
        expr  +  term
         |      / | \
       term  term  *  factor
         |     |        |
      factor factor     c
         |     |
         a     b
```

Every level of precedence gets its own layer of nonterminals. The parentheses, operators, and all grammar scaffolding are present.

## Abstract Syntax Tree (AST)

An AST strips away everything that was only needed for parsing and keeps just the essential structure — what the program *means*.

**Properties:**

- Nodes represent **operations** and **values**, not grammar rules.
- Syntactic sugar is gone: no parentheses nodes, no semicolon nodes, no nonterminals that only exist for precedence.
- The tree directly encodes the structure a compiler or interpreter needs to evaluate or transform the program.

**Example:** The same `a + b * c` becomes:

```
        +
       / \
      a    *
          / \
         b   c
```

Precedence is captured by the tree structure itself (`*` is deeper, so it evaluates first), not by extra nonterminal layers.

## Key Differences

| Aspect | Parse Tree | AST |
|---|---|---|
| **Purpose** | Shows how the grammar derived the input | Shows the semantic structure of the program |
| **Nodes** | Nonterminals and terminals from the grammar | Operators, literals, identifiers, statements |
| **Size** | Larger — includes all grammar scaffolding | Smaller — only meaningful structure |
| **Punctuation** | Present (parentheses, semicolons, commas) | Absent — already encoded in tree shape |
| **Grammar-dependent** | Yes — different grammars for the same language produce different parse trees | No — equivalent programs produce the same AST regardless of grammar |
| **Who uses it** | The parser (internally) | Everything after parsing: type checkers, optimizers, code generators, interpreters |

## Why the Distinction Matters

A parse tree for `(a + b)` and `a + b` would look different — the first has nodes for `(` and `)`. But their ASTs are identical, because the parentheses don't add meaning; they just override default precedence, which the tree structure already captures.

This is the whole point: **the AST is the interface between the parser and the rest of the compiler/interpreter**. It says "here's what the program means" without burdening downstream phases with how the grammar happened to be written.

## In Practice (Monkey Language / Pratt Parsing)

In the Monkey interpreter from *Writing an Interpreter in Go*, the parser builds an AST directly — it never constructs a full parse tree. Each Pratt parsing function (`parseExpression`, `parseIfExpression`, etc.) returns an AST node like `InfixExpression`, `CallExpression`, or `IfExpression`. The grammar rules are implicit in the parsing code; the output is purely semantic structure.

This is typical of hand-written parsers. Full parse trees are mainly useful in educational contexts and in parser generators that need to be grammar-agnostic.
